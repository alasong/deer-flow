# 审查员维度清单

每个维度在 spawn 时附带以下结构化检查清单，确保审查覆盖完整。

## correctness

- Plan 中的所有逻辑分支是否均已覆盖？
- 边界条件（空值、零值、越界）是否正确处理？
- 类型转换、数值运算是否有溢出或精度丢失风险？
- 并发控制：竞态条件、死锁、原子性是否保证？
- 状态机转换是否完整且无非法跳转？
- 错误路径是否留下不一致的中间状态？
- 外部依赖（API、DB、文件）的假设是否已验证？
- **richness | concrete_examples (P2)**: 每个抽象设计模式/规则是否有至少一个具体示例？标注缺少示例的抽象陈述。
- **richness | motivation_clarity (P2)**: 每个关键设计决策的 YAML rationale 是否清晰表达了为什么？标注理由含糊或缺失的决策。
- **richness | empirical_grounding (P3)**: 关键量化主张（性能目标、覆盖率、准确率）是否有行业数据或基准支撑？标注无支撑的数值主张。

## security

- 输入校验是否覆盖所有外部数据源（用户输入、文件、API参数）？
- 认证、鉴权逻辑是否覆盖所有入口点？
- 敏感数据（密钥、token、PII）是否未泄漏到日志/错误消息/响应中？
- 注入攻击（SQL、NoSQL、命令、XXE、LDAP）是否被防范？
- 会话管理：CSRF token、JWT 签名验证、会话固定保护？
- 文件操作：路径遍历、上传类型限制、权限检查？
- CORS、HTTPS 强制、安全头是否配置正确？

## performance

- 新增查询或 API 调用是否在循环内？是否需要批处理？
- 是否存在无界增长的数据结构或循环？
- 缓存策略：缓存键正确性、失效机制、命中率预估？
- 数据库查询：是否缺少索引导致全表扫描？N+1 问题？
- 资源管理：连接池、内存分配、goroutine/线程泄漏？
- 热路径上的序列化/反序列化开销是否合理？

## api_design

- 新增/修改的 API 是否向后兼容？若 breaking，是否有迁移计划？
- 命名是否符合项目约定（RESTful、gRPC、GraphQL schema）？
- 错误响应格式是否一致且包含足够诊断信息？
- 分页、排序、过滤接口是否遵循统一模式？
- 请求/响应模型是否包含必要的验证注解？
- 版本策略：URL、header、或参数版本化是否明确？

## test_quality

- 覆盖率报告是否有显著下降？关键路径是否有对应测试？
- 测试是否覆盖了边缘用例（空输入、超时、错误返回）？
- 测试断言是否验证了正确性而非仅检查无异常？
- mock/fake 的假设是否与真实行为一致？
- 集成测试是否覆盖了外部依赖的故障模式？
- 是否有回归测试用例覆盖之前修复的 bug？

## maintainability

- 新增代码是否遵循现有架构模式（分层、依赖注入、单一职责）？
- 是否有明显的代码重复可以通过抽象消除？
- 函数/方法长度是否在合理范围内？是否需要拆分？
- 命名是否清晰表达意图而非实现细节？
- 注释是否解释"为什么"而非"是什么"？
- 模块间耦合是否合理？是否需要接口抽象？

## reliability

- 所有外部调用是否有超时设置？超时值是否合理？
- 重试策略：是否指数退避？是否幂等？是否有熔断？
- 错误处理：是否有裸 panic/throw？错误是否被吞没或忽略？
- 关键路径是否有降级策略（circuit breaker、fallback）？
- 可观测性：关键操作是否有日志、指标、追踪？
- 持久化：事务边界是否完整？一致性和持久性是否保证？

## data_privacy

- 是否处理了个人数据（PII）？是否明确数据分类？
- 数据采集是否有合法依据并满足最小化原则？
- 日志中是否包含 PII（邮箱、IP、用户ID）？是否需要脱敏？
- 数据保留和删除策略是否明确？
- 数据传输是否加密（TLS/HTTPS）？存储是否加密？
- 是否满足相关法规（GDPR、CCPA、HIPAA、PCI-DSS）？
- 数据共享给第三方是否有合规审查？
- 用户数据导出/删除权利是否可实现？
