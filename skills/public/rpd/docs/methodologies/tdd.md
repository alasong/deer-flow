# TDD（测试驱动开发）

## 概述

先写测试。看它失败。写最少代码让它通过。

**核心原则：如果你没有亲眼看到测试失败，你就无法确定它测试的是正确的东西。**

**违背规则的"字面意思"等于违背了规则的精神。**

---

## 铁律（The Iron Law）

```
没有生产代码（Production Code）能在没有失败测试之前编写。
```

**在写测试之前就写了生产代码？删掉它。从头开始。**

**没有例外：**
- 不要留着"作参考"
- 不要在写测试时去"适配"它
- 不要去看它
- 删掉就是删掉

从测试出发重新实现。没得商量。

---

## 何时使用

**始终适用：**
- 新功能开发
- Bug 修复
- 重构
- 行为变更

**例外（需要与搭档确认）：**
- 一次性原型（throwaway prototype）
- 自动生成的代码
- 配置文件

想着"就这一次不写 TDD"？打住。这是自我合理化。

## 红-绿-重构循环（Red-Green-Refactor）

```
   ┌──────────┐
   │   RED    │  写一个失败的测试
   │ 写测试   │
   └────┬─────┘
        │
        ▼
   ┌──────────┐
   │  验证    │  ←─ 测试没失败？回到 RED
   │  RED     │     测试报错？修复后重试
   └────┬─────┘
        │  确认失败原因正确
        ▼
   ┌──────────┐
   │  GREEN   │  写最少代码让测试通过
   │ 实现     │
   └────┬─────┘
        │
        ▼
   ┌──────────┐
   │  验证    │  ←─ 测试没通过？修复代码，不是修复测试
   │  GREEN   │     其他测试失败？立即修复
   └────┬─────┘
        │  全部通过
        ▼
   ┌──────────┐
   │ REFACTOR │  保持绿色，优化代码
   │ 重构     │
   └────┬─────┘
        │
        ▼
   下一个测试 → RED
```

### RED — 写一个失败的测试

写一个最小化的测试，描述期望的行为。

**好的测试：**
```python
def test_retry_failed_operations_three_times():
    attempts = 0

    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise RuntimeError("fail")
        return "success"

    result = retry_operation(operation)

    assert result == "success"
    assert attempts == 3
```
清晰的命名，测试真实行为，一件事。

**坏的测试：**
```python
def test_retry_works():
    # 模糊的命名，测试 mock 而非代码
    mock = MagicMock(side_effect=[RuntimeError, RuntimeError, "success"])
    retry_operation(mock)
    assert mock.call_count == 3
```

**要求：**
- 一个测试只测一件事
- 清晰的名称
- 使用真实代码（除非不得已，否则不要用 mock）

### 验证 RED — 亲眼看到它失败

**强制步骤。绝不能跳过。**

```bash
python3 -m pytest path/to/test_file.py -q
```

确认：
- 测试确实 FAIL（不是 ERROR）
- 失败消息符合预期
- 失败是因为功能缺失（不是拼写错误）

**测试通过了？** 你测试的是已有行为。修正测试。

**测试报错了？** 修复错误，重新运行直到它正确失败。

### GREEN — 最少实现代码

写最简代码让测试通过。

**好的实现：**
```python
def retry_operation(fn):
    for i in range(3):
        try:
            return fn()
        except Exception:
            if i == 2:
                raise
```
刚好能通过。

**坏的实现：**
```python
def retry_operation(fn, max_retries=3, backoff=None, on_retry=None):
    # YAGNI — 过度设计，测试没有要求这些
    ...
```

不要额外加功能、重构其他代码、或"改进"超出测试要求的东西。

### 验证 GREEN — 看到它通过

**强制步骤。**

```bash
python3 -m pytest path/to/test_file.py -q
```

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误、无警告）

**测试没通过？** 修复代码，不是修复测试。

**其他测试失败了？** 立即修复。

### REFACTOR — 清理代码

仅在 GREEN 状态下做：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿色。不要添加新行为。

### 重复

为下一个功能写下一个失败的测试。

---

## 测试质量指南

| 质量维度 | 好的测试 | 坏的测试 |
|---------|---------|---------|
| **最小化** | 一件事。名字里有"and"就该拆分 | `test_validates_email_and_domain_and_whitespace` |
| **清晰** | 名字描述行为 | `test_test1` |
| **展示意图** | 展现期望的 API | 掩盖了代码应该做什么 |

---

## 为什么顺序如此重要

**"我之后会写测试来验证它工作"**

事后写的测试立即通过。立即通过证明不了任何事情：
- 可能测试的是错误的东西
- 可能测试的是实现细节，而非行为
- 可能遗漏了你忘记的边界情况
- 你从没见过它捕获 bug

测试先行强制你看到测试失败，证明它确实在测试某些东西。

**"我已经手动测试了所有边界情况"**

手动测试是 ad-hoc 的。你以为测试了所有，但实际上：
- 没有测试记录
- 代码变更后无法重新运行
- 压力下容易遗漏测试
- "我试过它能用"不等于全面覆盖

自动化测试是系统性的。每次以同样的方式运行。

**"删掉 X 小时的工作太浪费了"**

沉没成本谬误。时间已经花掉了。你现在的选择是：
- 删掉并用 TDD 重写（再花 X 小时，高置信度）
- 保留并事后补测试（30 分钟，低置信度，很可能有 bug）

真正的"浪费"是保留你无法信任的代码。没有真正测试就能工作的代码是技术债务。

**"TDD 太教条了，务实意味着灵活适配"**

TDD 本身就是务实的：
- 在提交前发现 bug（比事后调试快得多）
- 防止回归（测试立即捕获破坏）
- 文档化行为（测试展示了如何使用代码）
- 支持重构（自由修改，测试捕获破坏）

所谓"务实"的捷径 = 在生产环境调试 = 更慢。

**"事后补测试能达到同样的目标——重要的是精神而不是仪式"**

不是。事后测试回答"这段代码做了什么？" 测试先行回答"这段代码应该做什么？"

事后测试被你的实现偏见。你测试的是你构建的东西，而不是真正需要的东西。你验证的是你记得的边界情况，而不是发现的边界情况。

30 分钟事后测试 ≠ TDD。你得到了覆盖率，但失去了测试有效的证明。

## 常见自我合理化（Common Rationalizations）

| 借口 | 现实 |
|------|------|
| "太简单了，不需要测试" | 简单的代码也会出问题。写个测试只要 30 秒。 |
| "我之后再补测试" | 立即通过的测试证明不了任何事情。 |
| "事后测试能达到同样的目标" | 事后测试 = "代码做了什么？" 测试先行 = "代码应该做什么？" |
| "已经手动测试过了" | Ad-hoc ≠ 系统化。没有记录，无法重新运行。 |
| "删掉 X 小时的成果太浪费" | 沉没成本谬误。保留未经验证的代码就是技术债务。 |
| "留着作参考，先写测试" | 最终你一定会去适配它。那等于事后测试。删掉就是删掉。 |
| "需要先探索一下" | 可以。探索完后扔掉探索代码，从 TDD 开始。 |
| "测试难写 = 设计不清晰" | 倾听测试的声音。难测试 = 难使用。 |
| "TDD 会拖慢我" | TDD 比调试快得多。所谓务实 = 测试先行。 |
| "手动测试更快" | 手动无法证明边界情况。每次改动你都要重新测试。 |
| "已有代码没有测试" | 你在改进它。为已有代码补上测试。 |

## 红旗（Red Flags）— 立即停止并重新开始

- 在测试之前写了代码
- 实现之后再写测试
- 测试立即通过
- 说不清测试为什么失败
- "以后再加"测试
- 合理化"就这一次例外"
- "我已经手动测试过了"
- "事后测试能达到同样目的"
- "重要的是精神而不是仪式"
- "留着作参考" 或 "适配已有代码"
- "已经花了 X 小时，删掉太浪费"
- "TDD 太教条了，我这是务实"
- "这次情况不同因为……"

**以上任意一条出现 = 删除代码。从 TDD 重新开始。**

---

## 验证清单（Verification Checklist）

在标记完成之前：

- [ ] 每个新函数/方法都有测试
- [ ] 亲眼看到每个测试在实现前失败
- [ ] 每个测试因为预期原因失败（功能缺失，而非拼写错误）
- [ ] 写了最少代码让每个测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）
- [ ] 测试使用真实代码（mock 仅在不得已时使用）
- [ ] 边界情况和错误覆盖完整

无法勾选所有项目？你跳过了 TDD。重新开始。

## 卡住时的解决方案

| 问题 | 解决方案 |
|------|---------|
| 不知道如何测试 | 先写期望的 API。先写断言。与搭档讨论。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 到处都需要 mock | 代码耦合太紧。使用依赖注入。 |
| 测试设置代码太多 | 提取辅助函数。仍然复杂？简化设计。 |

---

## 测试反模式（Anti-Patterns）

- **测试 mock 而非真实行为** — mock 的行为可能与真实实现不同
- **为测试在生产类中添加方法** — 不要为了测试而污染生产接口
- **不理解依赖关系就 mock** — mock 了不该 mock 的东西，测试失去意义
- **测试实现细节** — 重构时测试会大面积碎裂，降低重构意愿
- **过度设置** — 测试的 setup 比测试本身还长，说明设计有问题

---

## 与调试的集成

发现 bug？先写一个能复现这个 bug 的失败测试。走完整 TDD 循环。这个测试既证明修复有效，又防止回归。

**绝不能在没有测试的情况下修复 bug。**

---

## 最终规则

```
有生产代码 → 必须有对应的测试，且该测试曾失败过
否则 → 不是 TDD
```

没有例外。除非与你的搭档确认同意。

## TDD 在 RPD 中的应用

当你在 RPD 树中的某个子节点选择 implement 模式，并应用 TDD 方法论时：

1. **创建测试文件**作为第一个 artifact（在节点工作目录中）
2. 运行 `rpd-engine.py tree node-start <id>` 开始工作节点
3. 遵循 Red → Green → Refactor 循环：
   - 每次只写一个测试
   - 亲眼验证测试失败（RED）
   - 写最少实现代码（GREEN）
   - 亲眼验证测试通过
   - 重构保持整洁
4. 每完成一个完整的 Red-Green-Refactor 循环，记录一条 `rpd decision_log`：
   - 记录测试了什么、遇到什么问题、最终方案
5. 完成所有功能后，验证清单通过后运行 `rpd-engine.py tree node-done <id>`

### 与 RPD 并行能力的配合

RPD 默认并行的特性意味着：多个 implement 子节点可以各自独立运行 TDD 循环。每个子节点拥有独立的测试套件和实现代码，互不干扰。这天然适合 TDD 的"一次只做一件事"原则——每个子节点聚焦一个具体功能，独立完成红-绿-重构。

### 注意事项

- TDD 更适用于有明确输入/输出的功能模块，不适合 UI 布局、协调整合等工作
- 对于探索性工作，先使用 spike 模式，清除探索产物后再用 TDD 实现
- 测试覆盖率的目的是**保障信心**，不是追求数字
- 从最核心的业务逻辑开始测试，而不是基础设施
