# Systematic Debugging（系统化调试）

## 核心原则：The Iron Law

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

随机修复浪费时且制造新 bug。快速补丁掩盖深层问题。**症状修复等于失败。**

如果你还没有完成 Phase 1（Root Cause Investigation），则不能提议任何 fix。

## 适用场景

- 任何测试失败、bug、非预期行为
- 性能问题、构建失败、集成问题
- **尤其需要**：时间紧迫时（紧迫让人想猜）、"just one quick fix" 似乎很明显时、已经试过多轮修复时

**不要跳过**：问题看似简单时（简单 bug 也有 root cause）、赶时间时（systematic 比 thrashing 更快）

## 四个阶段

你必须按顺序完成每个阶段，通过 gate 才能进入下一阶段。

### Phase 1: Root Cause Investigation（根因调查）

**在尝试任何 fix 之前：**

1. **仔细阅读错误信息**
   - 完整阅读 stack traces，记录行号、文件路径、error codes
   - 不要跳过 errors 或 warnings

2. **稳定复现**
   - 能否可靠触发？精确的复现步骤是什么？
   - 若非稳定复现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - `git diff`、recent commits、新依赖、配置变化、环境差异

4. **跨组件系统（Multi-Component）的证据收集**

   **当系统涉及多个组件时（例如 CI→build→signing, API→service→database）：**

   **在提议 fix 之前，添加诊断仪器：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传播
     - 检查每一层的状态

   运行一次收集证据，展示在哪里断裂
   然后分析证据，定位故障组件
   然后深入研究该组件
   ```

5. **追踪数据流**

   **当错误在调用栈深处时：**
   - 坏值从哪里起源？
   - 谁用坏值调用了当前函数？
   - 向上追踪直到找到源头
   - **在源头修复，而非在症状处**

**Gate:** 只有完成 Phase 1（理解 WHAT 和 WHY）才能进入 Phase 2。

### Phase 2: Pattern Analysis（模式分析）

**找到模式，再动手 fix：**

1. **寻找工作示例**
   - 在相同代码库中找到类似的、正常工作的代码
   - 什么正常工作的东西和被破坏的类似？

2. **对比参考实现**
   - 如果是在实现某个 pattern，完整阅读参考实现
   - 不要浏览，阅读每一行
   - 在应用之前完全理解 pattern

3. **识别差异**
   - 正常 vs 异常之间有何不同？
   - 列出每一个差异，无论多小
   - 不要假设 "这个不会影响"

4. **理解依赖**
   - 这个组件依赖哪些其他组件？
   - 需要什么 settings、config、environment？
   - 它做了哪些假设？

**Gate:** 只有完成 Phase 2（识别出差异）才能进入 Phase 3。

### Phase 3: Hypothesis and Testing（假设与验证）

**科学方法：**

1. **形成单一假设**
   - 清晰陈述："我认为 X 是根因，因为 Y"
   - 写下来，具体而非模糊

2. **最小化验证**
   - 做**最小可能的变更**来验证假设
   - 一次只改变一个变量
   - 不要同时修复多个问题

3. **验证后再继续**
   - 验证通过？→ 进入 Phase 4
   - 未通过？→ 形成新假设
   - **不要在已有变更上叠加更多 fix**

4. **当你不确定时**
   - 说 "我不理解 X"
   - 不要假装知道
   - 寻求帮助，深入研究

**Gate:** 只有确认假设（或形成新假设）后才能进入 Phase 4。

### Phase 4: Implementation（实现修复）

**修复根因，而非症状：**

1. **先写一个失败的测试**
   - 最简单的可复现用例
   - 尽可能写成自动化测试
   - 确保在 fix **之前**就有了
   - 这是 verification-first 原则

2. **实施单一修复**
   - 针对已确认的根因
   - **一次只改一个地方**
   - 不做 "趁机随手" 的改进
   - 不捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试？
   - 问题确实已解决？

4. **如果修复无效**
   - **停止**
   - 计数：已经尝试了多少次 fix？
   - 如果 < 3 次：回到 Phase 1，用新信息重新分析
   - **如果 ≥ 3 次：停止，质疑架构（见下方第 5 步）**
   - 不要在没有 architectural discussion 的情况下尝试第 4 个 fix

5. **3 次以上修复失败 → 质疑架构（Question Architecture）**

   **提示架构问题的模式：**
   - 每次修复都暴露出新的 shared state / coupling / 问题
   - 修复需要 "massive refactoring" 才能落地
   - 每个修复都导致其他地方出现新症状

   **停止并追问根本问题：**
   - 当前 pattern 从根本上说是否稳健？
   - 我们是否因为 "惯性" 而在坚持错误的模式？
   - 是否应该重构架构，而不是继续修复症状？

   **这不是假设验证失败 —— 这是错误的架构选择。**

## 红旗警示（Red Flags）—— 停止并回到流程

如果你发现自己有这些想法：

- "先 quick fix，之后再调查"
- "试试改 X 看看能不能跑"
- "一次改多处，统一跑测试"
- "跳过测试，我手动验证"
- "大概是因为 X，让我 fix 一下"
- "我不完全理解，但也许这样可以"
- "Pattern 说应该是 X，但我换个方式实现"
- "主要问题是：……"（直接列出 fix 而没有调查）
- 在追踪数据流之前就提议方案
- **"再试一次 fix"（已经试过 2 次以上）**
- **每次修复都出现新问题在不同的地方**

**所有这些都意味着：停止。回到 Phase 1。**

**如果 3+ 次修复失败：质疑架构（见 Phase 4.5）。**

## 常见合理化借口（Common Rationalizations）

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。对简单 bug 流程更快。 |
| "紧急情况，没时间走流程" | Systematic debugging 比 guess-and-check thrashing **更快**。 |
| "先试试这个，之后再调查" | 第一次 fix 定调子。从一开始就做对。 |
| "确认 fix 有效后再写测试" | 未经测试的 fix 不会持久。先写测试证明它。 |
| "一次改多处省时间" | 无法隔离哪个变更有效，还会制造新 bug。 |
| "参考太长，我按自己的理解改" | Partial understanding 保证引入 bug。完整阅读。 |
| "我看到问题了，让我 fix" | 看到症状 ≠ 理解根因。 |
| "再试一次 fix"（2 次后） | 3+ 次失败 = 架构问题。质疑 pattern，不要再 fix。 |

## 快速参考

| Phase | 关键活动 | 成功标准 |
|-------|---------|---------|
| **1. Root Cause** | 读错误、复现、查变更、收集证据 | 理解 WHAT 和 WHY |
| **2. Pattern** | 找工作示例、对比 | 识别差异 |
| **3. Hypothesis** | 形成理论、最小验证 | 确认假设或形成新假设 |
| **4. Implementation** | 写测试、fix、验证 | Bug 解决，测试通过 |

## 当流程揭示 "没有根因" 时

如果系统化调查后发现问题是环境相关、定时依赖或外部因素：

1. 你已完成流程
2. 记录已调查的内容
3. 实施合适的处理（retry、timeout、error message）
4. 添加 monitoring/logging 以便未来调查

**但：95% 的 "没有根因" 案例其实是不完整的调查。**

## Systematic Debugging 在 RPD 中的应用

- 在 RPD 树的 **C（Check）阶段**，如果遇到 evaluate/audit 模式的任务，选择本方法论
- Phase 1–4 可以映射到 RPD 子节点，每个 phase 一个子节点，按序执行
- 每完成一个 phase gate，记录一条 decision_log
- Phase 4 Step 1 的 "先写失败测试" 可以在 D 阶段子节点中与 TDD 方法论协同使用
